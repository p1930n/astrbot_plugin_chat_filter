# 违禁词命中后推送与周期文件计划

## 目标

为 `astrbot_plugin_chat_filter` 后续实现一条可执行的违规处理链路：

1. 监听群内成员发送消息。
2. 命中违禁词后，将命中记录写入持久化存储。
3. 对发送者直接执行禁言。
4. 同时撤回原消息。
5. 将违禁词内容通过“合并转发”的方式推送到绑定的推送群。
6. 按时间窗口定期向推送群发送文件，文件内容为该监听群在本窗口内的违禁词历史。

本计划只定义工程设计与验收合同，不假设 AstrBot 或具体平台已经提供禁言、撤回、合并转发、发文件 API。上述平台能力必须在实现前按 AstrBot 官方文档和目标平台官方能力确认。

## 命令建议

建议新增绑定命令：

```text
.cf bind [listening group] [push group]
```

语义：

- `listening group`：被监听的群聊 ID，违规事件发生在该群。
- `push group`：接收合并转发消息和周期历史文件的群聊 ID。
- 本插件按 QQ 机器人场景设计，默认监听群和推送群处于同一 QQ 平台上下文；命令参数只传 QQ 群号，`platform` 从当前 AstrBot 事件上下文取得。
- 不支持跨平台绑定；若后续需要跨平台推送，应另行设计平台标识、能力降级和权限边界。
- 同一个监听群同一时间只绑定一个推送群，重复绑定时覆盖旧绑定或显式要求先解绑，需在实现前确定交互策略。

建议补充但不强制本轮实现的运维命令：

```text
.cf bind list
.cf unbind [listening group]
.cf history [listening group] [time range]
```

权限建议：

- 绑定、解绑、查询绑定列表应只允许插件管理员或平台官方管理员权限用户执行。
- 权限来源只能通过 AstrBot 或平台官方上下文获取，不能用本地硬编码用户 ID 代替官方权限判断。

## 数据表建议

若现有持久化层已经有命中记录表，应优先扩展现有表结构，不并行新建重复语义的表。若需要新增表，建议如下。

### 群绑定表

表名建议：`chat_filter_group_bindings`

字段建议：

- `id`：主键。
- `listening_group_id`：监听群 ID，唯一索引。
- `push_group_id`：推送群 ID。
- `platform`：平台标识，从当前 QQ 适配器上下文取得，用于区分不同适配器下相同群号。
- `enabled`：绑定是否启用。
- `created_at`：创建时间。
- `updated_at`：更新时间。
- `created_by`：创建绑定的用户 ID，写入前需脱敏策略评估。

索引建议：

- `unique(platform, listening_group_id)`。
- `index(platform, push_group_id)`，用于后续排查或批量迁移。

### 违禁词命中记录表

表名建议：`chat_filter_violations`

字段建议：

- `id`：主键。
- `platform`：平台标识。
- `group_id`：命中发生的监听群 ID。
- `user_id`：触发用户 ID，展示和日志使用时需要脱敏。
- `message_id`：原消息 ID，用于撤回和审计；失败时保留原始 ID 可能有隐私风险，需控制日志输出。
- `matched_keyword`：命中的违禁词或规则标识。
- `matched_content`：命中的文本片段。该字段敏感，应考虑长度限制、脱敏展示、访问权限和保留周期。
- `raw_message_digest`：原消息摘要，不保存完整原文，便于去重和排查。
- `action_mute_status`：禁言执行状态，例如 `pending`、`success`、`failed`、`unsupported`。
- `action_recall_status`：撤回执行状态。
- `action_forward_status`：合并转发推送状态。
- `file_batch_id`：已归档到哪个周期文件批次，未发送时为空。
- `created_at`：命中时间。
- `updated_at`：状态更新时间。

索引建议：

- `index(platform, group_id, created_at)`，用于按群生成周期文件。
- `index(file_batch_id)`，用于避免重复发送同一窗口历史。
- `index(platform, message_id)`，用于幂等处理。

### 周期文件批次表

表名建议：`chat_filter_violation_batches`

字段建议：

- `id`：主键。
- `platform`：平台标识。
- `listening_group_id`：监听群 ID。
- `push_group_id`：推送群 ID。
- `window_start`：统计窗口开始时间。
- `window_end`：统计窗口结束时间。
- `record_count`：文件内记录数。
- `file_name`：发送文件名。
- `send_status`：发送状态，例如 `pending`、`success`、`failed`、`unsupported`。
- `error_code`：脱敏后的失败分类，不记录堆栈、token、cookie 或平台敏感响应。
- `created_at`：批次创建时间。
- `sent_at`：发送完成时间。

索引建议：

- `unique(platform, listening_group_id, window_start, window_end)`，避免重复生成批次。
- `index(send_status, created_at)`，用于失败重试或人工排查。

## 事件流程

### 绑定流程

1. 管理员发送 `.cf bind [listening group] [push group]`。
2. 入口层从 AstrBot 事件对象中取得调用者身份、平台标识和权限上下文。
3. 权限校验通过后，对两个 QQ 群号做类型、长度和格式校验；不接受跨平台目标。
4. 写入或更新绑定表。
5. 返回绑定结果，不在日志中输出完整用户隐私、token、cookie 或平台上下文。

### 命中处理流程

1. 群消息事件进入插件入口层。
2. 入口层仅提取原生业务字段：平台、群 ID、用户 ID、消息 ID、文本内容、发送时间。
3. 业务层确认该群是否存在启用的绑定。
4. 匹配违禁词，命中后先写入命中记录，记录状态为各动作 `pending`。
5. 调用平台能力执行禁言。
6. 调用平台能力撤回原消息。
7. 按绑定关系组装合并转发内容，并推送到推送群。
8. 分别更新禁言、撤回、合并转发的执行状态。
9. 任一动作失败都不应回滚已经入库的命中记录；失败状态用于后续审计和人工补偿。

处理顺序建议保持“先入库，后执行平台动作”，避免平台动作执行后本地无审计记录。

### 周期文件流程

1. 后台任务按配置的时间间隔扫描已绑定监听群。
2. 以固定时间窗口查询该监听群未归档或未发送的违禁词记录。
3. 生成文件内容，至少包含：命中时间、监听群、用户脱敏标识、违禁词或规则标识、命中文本片段、禁言状态、撤回状态、合并转发状态。
4. 文件内容必须避免写入 token、cookie、完整平台上下文、异常堆栈和 SQL。
5. 调用平台发文件能力，将文件发送到对应推送群。
6. 发送成功后更新批次和记录的 `file_batch_id`。
7. 发送失败时记录脱敏错误分类，保留批次 `failed` 状态，等待下一轮重试或人工处理。

文件生成位置建议使用插件受控数据目录或临时目录，并在发送成功后按配置清理。不能把运行时生成文件散落到源码目录。

## 失败降级

- 入库失败：停止后续禁言、撤回和推送动作，通过标准 logger 输出脱敏错误；不允许只执行平台动作但无本地记录。
- 禁言能力不可用：记录 `unsupported` 或 `failed`，继续尝试撤回和推送。
- 撤回能力不可用：记录 `unsupported` 或 `failed`，继续尝试合并转发推送。
- 合并转发能力不可用：降级方案必须先经产品确认；默认只记录 `unsupported`，不要擅自改成普通文本刷屏。
- 推送群未绑定：只写入命中记录并执行禁言、撤回；不执行推送和周期文件发送。
- 推送群发送失败：保留失败状态和重试条件，不重复执行禁言或撤回。
- 周期文件发送失败：批次保持 `failed`，下一轮可按批次幂等重试。
- 平台限流或超时：按 async runtime 设计加超时、限流和有限重试；重试耗尽后进入失败状态。
- 消息事件字段缺失：Fail-Fast 跳过本次处理，并输出脱敏日志，避免构造伪造 ID。

## 平台能力待确认点

实现前必须确认以下能力，不得臆造 API：

- AstrBot 当前目标版本的事件装饰器、事件对象字段、配置注入方式和标准响应装配方式。
- 如何从 AstrBot 或平台官方上下文获取群 ID、用户 ID、消息 ID、平台标识和管理员权限。
- QQ 适配器下监听群和推送群是否都能由机器人访问；绑定时是否需要主动校验机器人在推送群内且具备发消息、发文件和合并转发权限。
- 目标平台是否支持机器人直接禁言群成员，以及所需权限、禁言时长范围和失败错误码。
- 目标平台是否支持撤回群成员消息，以及撤回时间窗口、权限要求和失败错误码。
- AstrBot 或目标平台是否支持“合并转发”消息发送；如果支持，需要确认消息节点结构、发送接口、长度限制和敏感内容策略。
- AstrBot 或目标平台是否支持向群聊发送文件；如果支持，需要确认文件大小限制、文件名限制、临时文件生命周期和失败错误码。
- 热重载或插件卸载时，后台周期任务如何取消，数据库连接和临时文件如何释放。

## 验收清单

- 创建 `.cf bind [listening group] [push group]` 后，绑定关系可持久化，重载插件后仍可读取。
- 非管理员执行绑定命令时被拒绝，且日志不泄露完整平台上下文。
- 监听群成员发送违禁词后，命中记录先入库。
- 命中后尝试禁言发送者，成功与失败状态均会写回记录。
- 命中后尝试撤回原消息，成功与失败状态均会写回记录。
- 命中后按绑定关系向推送群发送合并转发消息，内容包含必要审计字段且不过量暴露原文。
- 未绑定推送群时，不执行合并转发和文件推送，但不影响命中记录、禁言和撤回。
- 周期任务能按时间窗口生成本群违禁词历史文件，并发送到绑定推送群。
- 周期文件发送成功后不会在下一轮重复发送同一窗口。
- 周期文件发送失败后有可审计状态，并可幂等重试。
- 插件热重载或卸载后，后台任务不会残留，临时文件不会无限增长。
- 数据库、平台 API、文件系统任一环节失败时，用户侧响应和日志均不泄露 token、cookie、SQL、堆栈或完整隐私内容。

## 隐私与日志边界

- 违禁词内容、命中文本、用户 ID、群 ID、消息 ID 都按敏感数据处理。
- 普通日志只记录脱敏 ID、动作状态和错误分类，不记录完整原文、token、cookie、平台上下文、SQL 或异常堆栈。
- 周期文件面向推送群可见，必须视为敏感审计材料；绑定推送群前应明确管理员权限和群可见范围。
- `matched_content` 建议做最大长度限制，超长内容截断并标记。
- 原消息全文不建议落库；若确需审计，应优先保存摘要和命中片段。
- 合并转发内容应只包含审计必要字段，避免把完整用户消息扩散到无关群。
- 历史文件应有保留周期和清理策略；发送成功后的本地临时文件应及时删除。
- 错误响应面向群聊时应简化为业务结果，不暴露内部路径、供应商错误详情或数据库结构。

## 实现边界

- 本计划不修改 `repository.py`、`models.py`、`main.py`、`settings.py`、`_conf_schema.json`。
- 后续实现涉及持久化表结构时，需要先按 persistence workflow 确认现有 repository 和 model 设计。
- 后续实现涉及 AstrBot 入口、事件对象、权限和响应装配时，需要先按 astrbot-integration workflow 确认官方 API。
- 后续实现涉及后台周期任务、平台异步 IO、重试、限流和热重载释放时，需要先按 async-runtime workflow 设计。

## 多推送群与群级周期补充

可行，而且你的补充把方案从“单推送群”升级成了更完整的审计系统。复杂度会增加，但不是不可控。

**1. 未绑定推送群仍处罚**

这个没问题，逻辑保持：

命中 -> 入库 -> 禁言 -> 撤回 -> 如果有推送群才推送。

推送绑定只影响“通知和周期报表”，不影响过滤处罚。

**2. 一个监听群绑定多个推送群**

可行，不会天然产生额外抽象层。SQLite 反而很适合这个关系。

不要做成一列里存：

```text
[listening group]:[push group1],[push group2]
```

这个展示形式可以给 WebUI 用，但数据库里不建议这样存。数据库应该一行一个绑定：

```text
group_push_bindings
- platform
- listening_group_id
- push_group_id
- enabled
- created_at
- updated_at
```

唯一约束：

```text
unique(platform, listening_group_id, push_group_id)
```

WebUI 展示时再聚合成：

```text
A群: 推送群1, 推送群2
B群: 推送群3
```

这样没有额外抽象层，只是正常的一对多关系。真正会增加复杂度的是“推送失败重试”：一个监听群绑定多个推送群后，每个推送群都要有独立成功/失败状态，不能一个失败影响全部。

**3. 禁言时间放 WebUI**

可行。建议 WebUI 放三个参数：

```text
mute_duration_seconds
mute_min_seconds
mute_max_seconds
```

但 QQ/适配器实际允许的最大最小值不要凭记忆硬编码，后续实现前按 AstrBot/QQ 适配器确认。运行时再 clamp 到允许范围。

**4. 周期文件按群自定义**

可行，但这里会明显增加调度复杂度。建议这样设计：

全局 WebUI 默认值：

```text
default_report_interval_days = 7
```

群级 SQLite 覆盖：

```text
group_report_policies
- platform
- listening_group_id
- interval_value
- interval_unit  # day/week/month
- next_run_at
- enabled
```

没有自定义的群就用默认 7 天。

这也不需要额外抽象层，但需要一个清晰的“调度服务”边界：它只负责查哪些群到时间、生成批次、发送文件、更新下次时间。不要把它塞进 `main.py` 或 matcher 里，否则耦合度会明显升高。

**5. 合并转发 + 单独日志消息**

可行，我建议按你说的做：

合并转发里放：

```text
发送人的原文片段
```

单独文本日志里放：

```text
某人违规
时间
监听群
QQ 号 / QQ 名称
禁言结果
撤回结果
推送时间
```

单独文本日志里不写具体违禁词。违禁词只在合并转发内容里体现。

这里要注意：QQ 名称是展示信息，不稳定，建议入库时保存一个 `sender_display_name_snapshot`，同时保留 `user_id` 作为审计主键。日志展示用 QQ 名称，追踪用 user_id。

结论：计划可行。建议下一步把计划文档改成新版结构：多推送群绑定、群级报表周期、禁言时间 WebUI 参数、合并转发和单独日志消息的内容边界。
